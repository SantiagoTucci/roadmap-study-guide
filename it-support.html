Fundamentos de IT

modelo
Modelo OSI & TCP/IP
La base conceptual de cómo viajan los datos en una red. Indispensable para diagnosticar problemas.
7 capas OSI4 capas TCP/IP
El modelo OSI tiene 7 capas: Física (cables, señales), Enlace (MAC, switches), Red (IP, routers), Transporte (TCP/UDP, puertos), Sesión, Presentación (TLS, cifrado), Aplicación (HTTP, DNS, SMTP).

En la práctica usás el modelo TCP/IP de 4 capas. El truco para soporte: cuando hay un problema de conectividad, bajás capa por capa. 
# Diagnóstico capa por capa
ping 192.168.1.1        # capa 3 (red) - ¿hay ruta?
ping google.com         # capa 3 + DNS
telnet servidor 443     # capa 4 (transporte) - ¿está el puerto abierto?
curl -v https://sitio   # capa 7 (aplicación) - ¿responde HTTP?
Si el ping a la IP funciona pero no al nombre, el problema es DNS. Si no llega ni la IP, es enrutamiento o capa física.

protocolos
DNS, DHCP & servicios core
Los servicios de red que toda infraestructura usa. Si fallan, todo falla.
DNS · DHCP · NTPSMTP · LDAP
DNS: traduce nombres a IPs. Registros clave: A (nombre→IPv4), AAAA (→IPv6), MX (correo), CNAME (alias), PTR (reverso), SRV (servicios como AD).

DHCP: asigna IPs dinámicamente. El proceso DORA: Discover → Offer → Request → Acknowledge. El lease tiene tiempo de vida. 
# Diagnóstico DNS en Windows
nslookup servidor.dominio.local
nslookup -type=MX empresa.com
Resolve-DnsName -Name dc01.corp.local -Type A

# Diagnóstico DNS en Linux
dig @8.8.8.8 empresa.com A
dig -x 192.168.1.1   # resolución inversa

# Ver lease DHCP actual (Windows)
ipconfig /all
ipconfig /release && ipconfig /renew
NTP: sincronización de tiempo. En Active Directory es crítico: si los relojes difieren más de 5 minutos, Kerberos rechaza la autenticación.
ver más ↓
hardware
Hardware de servidor
Componentes físicos de servidores, storage y networking. Para entender qué falla y cómo escalar.
CPU · RAM · RAIDHBA · NIC · iDRAC
Componentes que vas a ver en el trabajo:

iDRAC / iLO / IPMI: gestión remota fuera de banda. Permite encender, apagar, ver consola y logs del servidor aunque el SO no responda.
HBA (Host Bus Adapter): tarjeta de fibra óptica para conectar servidores a SANs.
RAID: combinación de discos para redundancia o performance. 
# Niveles de RAID más comunes:
RAID 0  → striping, máxima velocidad, sin redundancia
RAID 1  → espejo, tolera pérdida de 1 disco
RAID 5  → paridad distribuida, tolera 1 disco, mínimo 3
RAID 6  → tolera 2 discos, mínimo 4
RAID 10 → espejo + striping, alto rendimiento y redundancia
En servidores modernos el RAID lo gestiona un controlador (ej: PERC en Dell). Siempre revisar el estado de los discos en el portal de gestión.

Sistemas operativos

microsoft
Windows Server
El sistema operativo central del entorno. Gestión de roles, servicios, eventos y administración remota.
Server 2019/2022roles · features · WinRM
Los roles más comunes que vas a administrar: AD DS (Active Directory), DNS Server, DHCP Server, File Server, IIS (web), Hyper-V, RDS (Escritorio Remoto).

Administración remota con PowerShell: 
# Conectarse remotamente a un servidor
Enter-PSSession -ComputerName SERVER01
# O ejecutar un comando remoto sin abrir sesión interactiva
Invoke-Command -ComputerName SERVER01 -ScriptBlock { Get-Service }

# Ver servicios críticos
Get-Service | Where-Object {$_.Status -eq 'Stopped' -and $_.StartType -eq 'Automatic'}

# Ver logs de eventos del sistema
Get-EventLog -LogName System -Newest 50 -EntryType Error,Warning

# Ver uso de disco
Get-PSDrive | Where-Object {$_.Provider -like '*FileSystem*'}
El Visor de Eventos (Event Viewer) es tu mejor herramienta de diagnóstico. Los IDs de evento más importantes: 4625 (login fallido), 4624 (login exitoso), 1000 (error de aplicación), 6008 (apagado inesperado).

RDP & SSH
Los dos protocolos de acceso remoto más usados. RDP para Windows, SSH para Linux y equipos de red.

Active Directory

estructura
Estructura de AD DS
El directorio que gestiona usuarios, equipos y políticas en entornos Windows corporativos.
dominio · bosque · OUDC · schema · catálogo
Conceptos clave:
Dominio: unidad administrativa (ej: corp.empresa.com).
Bosque (Forest): colección de dominios con esquema compartido. Límite de seguridad real.
OU (Organizational Unit): contenedor para organizar objetos y aplicar GPOs de forma granular.
DC (Domain Controller): servidor que almacena y replica la base de datos de AD.
FSMO roles: cinco roles especiales de AD (PDC Emulator, RID Master, Infrastructure Master, Schema Master, Domain Naming Master). 
# Comandos PowerShell esenciales para AD

# Buscar usuario

  -AccountPassword (Read-Host -AsSecureString) -Enabled $true

# Desbloquear cuenta
Unlock-ADAccount -Identity jperez

# Ver DCs del dominio
Get-ADDomainController -Filter *

políticas
Group Policy (GPO)
Configuración centralizada de seguridad y comportamiento para usuarios y equipos del dominio.
GPMC · RSoP · herenciaComputer · User config
Las GPOs se aplican en este orden (LSDOU): Local → Sitio → Dominio → OU. Las GPOs de OU más específica ganan sobre las del dominio.

Usos comunes de GPOs: mapear unidades de red, configurar wallpaper corporativo, deshabilitar USB, instalar software, configurar políticas de contraseñas, redirigir carpetas de usuario.
# Forzar actualización de GPO en un equipo
gpupdate /force

# Ver GPOs aplicadas a un usuario/equipo
gpresult /R           # en consola
gpresult /H report.html  # reporte HTML completo

# Desde PowerShell remotamente
Invoke-Command -ComputerName PC001 -ScriptBlock { gpupdate /force }

# Ver qué GPO está aplicando una configuración específica (RSoP)
# Herramienta: gpmc.msc → Group Policy Results Wizard
Cuando una GPO no aplica, revisar: bloqueo de herencia, filtros de seguridad, WMI filters y si el equipo/usuario está en la OU correcta.

autenticación
Kerberos & LDAP
Los protocolos de autenticación y consulta de directorio que usa Active Directory internamente.
TGT · TGS · KDCLDAP 389 · LDAPS 636
Kerberos: protocolo de autenticación sin enviar contraseñas por la red. Flujo simplificado:
1. El usuario se autentica al KDC (el DC) y recibe un TGT (Ticket Granting Ticket).
2. Con el TGT pide un TGS (Ticket for a Service) para acceder a un recurso específico.
3. Presenta el TGS al servidor de destino.

LDAP: protocolo para consultar y modificar el directorio. Lo usan casi todos los sistemas corporativos para autenticar usuarios contra AD. 
# Consultar AD con LDAP desde PowerShell
$searcher = [adsisearcher]"(sAMAccountName=jperez)"
$searcher.FindOne()

# Con ldapsearch en Linux
ldapsearch -H ldap://dc01.corp.local -D "jperez@corp.local" `
  -W -b "DC=corp,DC=local" "(sAMAccountName=jperez)"

# Ver tickets Kerberos actuales (Windows)
klist
Problema común: "The trust relationship between this workstation and the primary domain failed" → el canal seguro entre el equipo y el DC se rompió. Solución: Test-ComputerSecureChannel -Repair o retirar y volver a unir al dominio.

Redes avanzadas

segmentación
VLANs & switching
Segmentar la red lógicamente sobre la misma infraestructura física. Seguridad y organización.
802.1Q · trunk · accessSTP · LACP
Una VLAN es una red lógica separada sobre la misma red física. Cada VLAN tiene su propio dominio de broadcast.

Puerto Access: pertenece a una sola VLAN (para equipos de usuario).
Puerto Trunk: transporta tráfico de múltiples VLANs con etiquetas 802.1Q (para uplinks entre switches y a routers/firewalls). 
# Configurar VLAN en switch Cisco (ejemplo)
vlan 10
 name USUARIOS
vlan 20
 name SERVIDORES
vlan 30
 name GESTION

# Puerto de usuario (access)
interface FastEthernet0/1
 switchport mode access
 switchport access vlan 10

# Puerto trunk (hacia otro switch o router)
interface GigabitEthernet0/1
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
STP (Spanning Tree): evita loops en redes con redundancia. Si hay un loop de red, la velocidad cae a 0 y el tráfico colapsa. STP apaga puertos redundantes hasta que hay un fallo.
ver más ↓
conectividad
VPN & acceso remoto
Extender la red corporativa de forma segura a usuarios remotos y sedes remotas.
IPSec · SSL/TLSsite-to-site · split tunnel
VPN site-to-site: conecta dos oficinas permanentemente. Usa IPSec. El tráfico entre las dos redes va cifrado.
VPN de usuario: permite a usuarios remotos conectarse a la red corporativa. Tecnologías comunes: OpenVPN, FortiClient, GlobalProtect (Palo Alto), Always On VPN de Microsoft.
Split tunnel: solo el tráfico corporativo va por la VPN, el tráfico a internet sale directo. Reduce carga pero requiere buena política de seguridad. 
# Diagnóstico de VPN en Windows
# Ver interfaces de red (incluye adaptadores VPN)
Get-NetAdapter | Select Name, Status, LinkSpeed

# Ver rutas (para entender si el tráfico va por VPN)
route print
# O
Get-NetRoute | Where-Object {$_.DestinationPrefix -ne "0.0.0.0/0"}

# Verificar conectividad a recursos internos
Test-NetConnection -ComputerName servidor-interno -Port 443
ver más ↓
firewall
Firewall & routing
Controlar qué tráfico pasa entre segmentos de red. Firewalls de próxima generación (NGFW).
ACL · NAT · NGFWFortinet · Palo Alto
Un firewall evalúa cada paquete contra reglas: origen, destino, puerto, protocolo. En NGFWs también puede inspeccionar el contenido (DPI).

NAT (Network Address Translation): traduce IPs privadas a una IP pública para salir a internet. PAT (Port Address Translation) permite que muchos equipos compartan una sola IP pública.
# Ver tabla de ruteo en Linux
ip route show
# Agregar ruta estática
ip route add 10.20.0.0/16 via 192.168.1.1

# Verificar si un puerto está alcanzable
telnet 192.168.1.100 443
# Alternativa moderna sin telnet
Test-NetConnection -ComputerName 192.168.1.100 -Port 443  # Windows
nc -zv 192.168.1.100 443  # Linux

# Ver reglas de firewall de Windows
Get-NetFirewallRule | Where-Object {$_.Enabled -eq "True"} | Select DisplayName
Cuando algo no llega: verificar en cada salto si hay bloqueo. El comando tracert (Windows) o traceroute (Linux) muestra cada hop y la latencia.

Seguridad básica

permisos
Permisos NTFS & RBAC
Controlar quién accede a qué en Windows. Permisos de archivos y roles de administración.
ACL · herencia · efectivoscompartidos · RBAC
Los permisos NTFS son locales al sistema de archivos. Los permisos de compartido (Share) son para acceso de red. El permiso efectivo es el más restrictivo entre ambos.
# Ver permisos NTFS de una carpeta
Get-Acl "C:\Datos\RRHH" | Format-List

# Asignar permiso a un grupo en una carpeta
$acl = Get-Acl "C:\Datos\RRHH"
$rule = New-Object System.Security.AccessControl.FileSystemAccessRule(
    "CORP\GRP_RRHH", "Modify", "ContainerInherit,ObjectInherit", "None", "Allow"
)
$acl.AddAccessRule($rule)
Set-Acl "C:\Datos\RRHH" $acl

# Ver permisos efectivos de un usuario sobre una carpeta
# Herramienta: propiedades de carpeta → Seguridad → Avanzado → Acceso efectivo
Principio de mínimo privilegio: cada usuario y servicio debe tener solo los permisos que necesita para su función, nada más. Es el principio de seguridad más importante en el día a día.
ver más ↓
identidad
MFA & gestión de contraseñas
Autenticación multifactor y políticas de contraseñas como primera línea de defensa.
TOTP · FIDO2 · passkeysFine-grained PSO
MFA: requerir algo que sabés (contraseña) + algo que tenés (teléfono/token) + algo que sos (biometría). Reduce drásticamente el riesgo de compromiso de cuentas.

En entornos Microsoft el MFA se gestiona con Azure AD / Entra ID y el Microsoft Authenticator. Para cuentas privilegiadas también existe PAM (Privileged Access Management).

Políticas de contraseñas con Fine-grained Password Policy en AD: permite tener políticas distintas para distintos grupos. Ej: usuarios normales → 8 chars, 90 días; admins → 16 chars, 30 días. 
# Ver política de contraseñas del dominio
Get-ADDefaultDomainPasswordPolicy

# Ver PSO (políticas granulares) aplicadas a un usuario
Get-ADUserResultantPasswordPolicy -Identity jperez

parches
Gestión de parches & antivirus
Mantener los sistemas actualizados es la defensa más efectiva contra vulnerabilidades conocidas.
WSUS · SCCM · IntuneDefender · EDR
WSUS: servidor local que centraliza las actualizaciones de Windows. Los equipos del dominio las descargan de WSUS en vez de Microsoft directamente (menos tráfico, más control).
SCCM/MECM: gestión completa de endpoints (parches, software, inventario, OSD).
Intune: versión cloud de gestión de dispositivos (MDM/MAM).

Virtualización

vmware
VMware vSphere
La plataforma de virtualización empresarial más usada. ESXi como hipervisor, vCenter para gestión centralizada.
ESXi · vCenter · vMotionHA · DRS · datastores
Arquitectura: ESXi es el hipervisor (tipo 1, corre directamente en hardware). vCenter es la consola de gestión centralizada para múltiples hosts ESXi.

Conceptos clave:
vMotion: migrar una VM encendida de un host a otro sin downtime.
Storage vMotion: mover los archivos de una VM a otro datastore en caliente.
HA (High Availability): reinicia automáticamente las VMs en otro host si el host falla.
DRS (Distributed Resource Scheduler): balancea la carga automáticamente entre hosts.
Snapshot: punto de restauración de una VM (NO es backup, tiene impacto en performance si se deja mucho tiempo). 
# Gestión con PowerCLI (VMware PowerShell)
Connect-VIServer -Server vcenter.corp.local

# Ver todas las VMs y su estado
Get-VM | Select Name, PowerState, NumCpu, MemoryGB

# Ver VMs con snapshots viejos (peligroso para performance)
Get-VM | Get-Snapshot | Where-Object {$_.Created -lt (Get-Date).AddDays(-7)}

# Migrar VM a otro host
Move-VM -VM "MI-SERVER" -Destination (Get-VMHost "esxi02.corp.local")
ver más ↓
proxmox
Proxmox VE
Plataforma de virtualización open source. Soporte para VMs (KVM) y contenedores (LXC). Muy usada en ambientes más chicos.
KVM · LXC · Cephcluster · live migration
Proxmox usa KVM para VMs completas y LXC para contenedores ligeros. Tiene interfaz web y API REST. Es gratuito (sin soporte) o con suscripción de empresa.
# Gestión desde CLI en el host Proxmox

# Iniciar/detener/reiniciar una VM

# Para contenedores LXC

pct start 200
pct enter 200  # ingresar al contenedor
Proxmox puede formar un cluster con múltiples nodos y usar Ceph para storage distribuido, logrando HA similar a VMware pero sin costos de licencia.

El hipervisor de Microsoft. Integrado en Windows Server, muy común en entornos que ya tienen stack Microsoft.

Microsoft Azure

identidad
Azure AD / Entra ID
El directorio de identidades en la nube de Microsoft. Base de la autenticación en todo el ecosistema Microsoft 365 y Azure.
SSO · MFA · SSPRConditional Access · PIM
Azure AD (ahora llamado Microsoft Entra ID) es diferente al AD on-premises: es un directorio cloud, no usa Kerberos, usa OAuth2/OIDC y SAML.

Conditional Access: políticas que permiten/bloquean acceso según condiciones (ubicación, dispositivo, riesgo de login).
PIM (Privileged Identity Management): activa roles privilegiados solo cuando se necesitan, con aprobación y justificación.

Azure VMs & recursos
Crear y gestionar máquinas virtuales, redes virtuales, storage y grupos de recursos en Azure.
VNet · NSG · ARMAvailability Sets · Scale Sets

Resource Group: contenedor lógico para recursos relacionados (práctica: un RG por proyecto/entorno).

storage

Almacenamiento en la nube: Blob, Files, Queues. Y backup con Azure Backup vault.
Blob · Files · LRS/GRSRecovery Services vault
Tipos de storage en Azure:
Blob: objetos no estructurados (imágenes, backups, logs, archivos grandes).

Amazon AWS

cómputo
EC2 & redes en AWS
Instancias de cómputo en AWS y su estructura de red: VPC, subnets, security groups.
EC2 · AMI · EBSVPC · SG · IGW
Conceptos clave de red en AWS:
VPC: red virtual privada en AWS, equivalente a una VNet de Azure.

Security Group: firewall a nivel de instancia (stateful). NACL: firewall a nivel de subnet (stateless).

almacenamiento
S3 & RDS
S3 para almacenamiento de objetos, RDS para bases de datos relacionales gestionadas.
S3 buckets · lifecycleRDS · Multi-AZ · read replica
S3: almacenamiento de objetos con 99.999999999% (11 nueves) de durabilidad. Tiene clases de almacenamiento: Standard, Intelligent-Tiering, Glacier (para archivos).
RDS: bases de datos gestionadas (MySQL, PostgreSQL, SQL Server, Oracle). AWS se encarga de backups, parches y failover.

Backup & continuidad

estrategia
RTO, RPO & regla 3-2-1
Los conceptos fundamentales de cualquier plan de continuidad del negocio.
RTO · RPO · SLA3-2-1 · DR · BCP
RPO (Recovery Point Objective): cuánta pérdida de datos es aceptable. Si hacés backup cada 24hs, tu RPO es 24hs de datos.
RTO (Recovery Time Objective): cuánto tiempo puede estar el sistema fuera de servicio. Si el negocio tolera 4hs de downtime, tu RTO es 4hs.

Regla 3-2-1:
— 3 copias de los datos
— en 2 tipos de medios distintos
— 1 copia offsite (en otra ubicación física o cloud)

BCP (Business Continuity Plan): plan completo que incluye procesos, personal y comunicaciones, no solo la tecnología.
DRP (Disaster Recovery Plan): el subconjunto de BCP enfocado en recuperar sistemas IT.

Un backup que nunca fue probado no es un backup real. Los restore tests periódicos son obligatorios.

herramienta
Veeam Backup
La solución de backup más usada para entornos virtualizados. Soporta VMware, Hyper-V y cloud.
jobs · retention · dedupinstant recovery · replication
Veeam trabaja con snapshots del hipervisor para hacer backups de VMs sin agente y sin downtime.

Conceptos clave:
Backup Job: tarea programada que respalda VMs a un repositorio.
Instant Recovery: levantar una VM directamente desde el backup en minutos, sin esperar a restaurar los datos (los lee del repositorio mientras corre).
Replication: mantiene una réplica de la VM en otro sitio para DR.
Sure Backup: verifica automáticamente que los backups son recuperables arrancando la VM en un entorno aislado.

Retención típica: backups diarios por 14 días, semanales por 4 semanas, mensuales por 12 meses. Ajustar según RPO del negocio.

replicación
Replicación & HA
Alta disponibilidad a nivel de datos y servicios. Diferencia entre backup, replicación y clustering.
sync · async · failoverclustering · load balancing
Backup: copia puntual en el tiempo, no continua. Sirve para recuperar datos borrados o corruptos.
Replicación: copia continua de datos a otro sitio. No protege de borrado accidental (se replica también el error).
Clustering (HA): múltiples servidores activos al mismo tiempo, el servicio no cae si uno falla.

Tecnologías de HA comunes:
Windows Failover Cluster: para SQL Server, File Server, Hyper-V en Windows.
Always On Availability Groups: HA nativa de SQL Server con replicación sincrónica.
DFS-R: replicación de file shares entre servidores Windows (Distributed File System Replication).

Resolución de incidentes

metodología
Triage & priorización
Evaluar rápidamente el impacto y urgencia de un incidente para actuar en el orden correcto.
impacto · urgenciamatriz de prioridad
Cuando llega un incidente, las primeras preguntas son:
¿Cuántos usuarios están afectados?
¿Qué sistema o servicio está caído?
¿Es un sistema crítico de negocio?
¿Tiene workaround disponible?

La prioridad = f(impacto, urgencia). Un usuario sin internet tiene urgencia alta pero bajo impacto. Un servidor de base de datos caído tiene ambos altos → P1.

Al atender un incidente siempre comunicar: a) que lo estás atendiendo, b) cada avance relevante, c) la resolución con causa y acciones tomadas. La comunicación proactiva reduce la presión sobre el equipo.

diagnóstico
RCA & post-mortem
Análisis de causa raíz después de un incidente grave. Para que no vuelva a ocurrir.
5 Whys · Ishikawablameless · action items
Un post-mortem blameless no busca culpables, busca causas sistémicas y mejoras. Nadie se equivoca a propósito; los sistemas fallan por falta de controles.

Estructura de un post-mortem:
1. Resumen: qué pasó, cuánto duró, impacto.
2. Línea de tiempo: secuencia exacta de eventos.
3. Causa raíz: los "5 Whys" para llegar a la causa real, no al síntoma.
4. Action items: tareas concretas con responsable y fecha para prevenir recurrencia.

Runbook: documento paso a paso para responder a incidentes conocidos. Reduce el tiempo de resolución y permite que cualquier miembro del equipo atienda el incidente sin depender de un experto específico.

# Script práctico: reporte de servidores con poco espacio en disco
$servidores = @("SRV01","SRV02","SRV03","SRV04")

$resultado = foreach ($server in $servidores) {
    try {
        $discos = Invoke-Command -ComputerName $server -ScriptBlock {
            Get-PSDrive -PSProvider FileSystem |
            Where-Object {$_.Free -lt 10GB} |
            Select-Object Name,
              @{N='LibreGB';E={[math]::Round($_.Free/1GB,2)}},
              @{N='TotalGB';E={[math]::Round(($_.Used+$_.Free)/1GB,2)}}
        }
        [PSCustomObject]@{Servidor=$server; Discos=$discos}
    } catch {
        [PSCustomObject]@{Servidor=$server; Error=$_.Exception.Message}
    }
}

$resultado | Export-Csv -Path ".\reporte-disco.csv" -NoTypeInformation

#!/bin/bash
# Script: backup diario de directorio con rotación de 7 días

ORIGEN="/var/www/html"
DESTINO="/backup"
FECHA=$(date +%Y%m%d_%H%M%S)
ARCHIVO="backup_web_$FECHA.tar.gz"
DIAS_RETENCION=7
LOG="/var/log/backup.log"

echo "$(date '+%Y-%m-%d %H:%M:%S') - Iniciando backup" >> $LOG

# Crear backup comprimido
if tar -czf "$DESTINO/$ARCHIVO" "$ORIGEN" 2>> $LOG; then
    echo "$(date '+%Y-%m-%d %H:%M:%S') - Backup OK: $ARCHIVO" >> $LOG
else
    echo "$(date '+%Y-%m-%d %H:%M:%S') - ERROR en backup" >> $LOG
    exit 1
fi

# Eliminar backups viejos
find "$DESTINO" -name "backup_web_*.tar.gz" \
  -mtime +$DIAS_RETENCION -delete

echo "$(date '+%Y-%m-%d %H:%M:%S') - Limpieza completa" >> $LOG

Hiperconvergencia

concepto
Infraestructura hiperconvergente
Combinar cómputo, storage y red en nodos estandarizados que se escalan horizontalmente.
Nutanix · vSAN · Azure Stackscale-out · SDS
La infraestructura tradicional tiene servidores, SANs y switches separados. La HCI (Hyper-Converged Infrastructure) pone todo en nodos commodity que se pueden agregar linealmente.

Nutanix AOS: la plataforma HCI más conocida. El hipervisor puede ser AHV (nativo), ESXi o Hyper-V. El storage es distribuido entre todos los nodos (Nutanix Distributed Storage Fabric).
VMware vSAN: storage distribuido nativo de VMware. Usa los discos locales de los hosts ESXi como storage compartido.
Ventajas: simplificación operativa, escalado lineal, eliminación de SAN dedicada.
Desventajas: vendor lock-in, costo inicial alto, menos flexibilidad que arquitectura tradicional para cargas muy específicas.

Seguridad avanzada

arquitectura
Zero Trust
Nunca confiar, siempre verificar. El modelo de seguridad moderno que reemplaza al perímetro de red.
identity-first · microseg.ZTNA · Conditional Access
El modelo tradicional asumía que todo dentro de la red era confiable. Zero Trust asume que la red ya está comprometida.

Principios de Zero Trust:
— Verificar explícitamente: autenticar y autorizar en cada request (MFA, device compliance, location).
— Mínimo privilegio: acceso just-in-time y just-enough-access (JIT/JEA).
— Asumir brecha: segmentar la red para limitar el movimiento lateral del atacante.

ZTNA (Zero Trust Network Access): reemplaza a la VPN tradicional. El acceso se otorga por aplicación, no por red. Ejemplos: Microsoft Entra Private Access, Cloudflare Access, Zscaler Private Access.

En Microsoft: Conditional Access + Intune compliance + Entra ID forma una buena base de Zero Trust.

Caminos de crecimiento
Desde soporte técnico hay múltiples rutas de especialización. Elegir según intereses.
Cloud Engineer · DevOpsSysAdmin · SRE · SecOps
Caminos posibles desde este rol:

Cloud Engineer: especialización en AWS/Azure. Diseñar, implementar y optimizar infraestructura cloud. Certs: AZ-104, SAA-C03, AZ-305 (Azure Solutions Architect).

DevOps/SRE: automatización, CI/CD, IaC, observabilidad. Herramientas: Terraform, Ansible, Kubernetes, GitLab CI, monitoring. Requiere más conocimiento de desarrollo.

SysAdmin / Infra Senior: profundizar en Windows Server, virtualización, storage, redes. El camino más directo desde el rol actual.

SecOps / Blue Team: seguridad operacional, SIEM, respuesta a incidentes, hardening. Certs: CompTIA Security+, SC-200, CISSP a largo plazo.

El primer año es de explorar. El segundo es de elegir dirección y profundizar. Las certificaciones dan estructura al aprendizaje, pero la experiencia práctica es lo que consigue el trabajo.

trabajo en equipo
Comunicación & trabajo en equipo
Las habilidades técnicas te consiguen el trabajo. Las blandas te hacen crecer dentro de él.
on-call · escaladaexpectativas · empatía
En soporte técnico las habilidades blandas son tan importantes como las técnicas porque trabajás directamente con usuarios frustrados y bajo presión.

Gestión de expectativas: comunicar proactivamente el estado. "Estoy trabajando en eso, te aviso en 30 minutos" reduce la presión mucho más que silencio.

Escalada a tiempo: saber cuándo escalas es tan importante como intentar resolver. Quedarse bloqueado sin pedir ayuda por orgullo tiene un costo alto.

On-call / guardia: en infraestructura es común rotar en guardias de disponibilidad. Requiere disciplina para documentar runbooks y dejar el entorno en condiciones para quien sigue.

Trabajo bajo presión: durante un incidente P1, la calma y el pensamiento sistemático valen más que la velocidad. Respirar, tomar notas, seguir el proceso.
