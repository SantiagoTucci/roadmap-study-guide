Fundamentos de Java

lenguaje
Sintaxis y tipos
Las bases del lenguaje: tipos primitivos, operadores, control de flujo y arrays.
int · boolean · String if · for · switch
Java diferencia entre tipos primitivos (viven en el stack) y objetos (viven en el heap). Los primitivos son: int, long, double, float, boolean, char, byte, short. 
int edad = 25;
String nombre = "Ana";   // objeto, no primitivo
boolean activo = true;
 
// autoboxing: Java convierte automáticamente
Integer edadObj = edad;  // int → Integer
El switch moderno (Java 14+) es más limpio y evita el bug del fall-through accidental. Los arrays tienen tamaño fijo; para tamaño dinámico usás ArrayList.
ver más ↓
POO
Programación orientada a objetos
Clases, herencia, polimorfismo, encapsulamiento e interfaces. La base de todo el ecosistema Java.
extends · implements abstract · interface
Los cuatro pilares: encapsulamiento (getters/setters, modificadores de acceso), herencia (extends), polimorfismo (un objeto puede comportarse como su padre o interfaz), abstracción (interfaces y clases abstractas). 
public interface Vehiculo {
    void arrancar();
    default void detener() { /* impl por defecto */ }
}
 
public class Auto extends Vehiculo {
    private String marca;  // encapsulamiento
 
    @Override
    public void arrancar() {
        System.out.println("Arrancando " + marca);
    }
}
Preferí composición sobre herencia. La herencia crea acoplamiento fuerte; las interfaces dan flexibilidad.
ver más ↓
colecciones
Collections Framework
List, Map, Set y Queue. Elegir la estructura correcta impacta directamente el rendimiento.
ArrayList · HashMap HashSet · TreeMap
List: ArrayList → acceso O(1), inserciones lentas al medio. LinkedList → inserciones O(1), acceso O(n).
Map: HashMap → búsqueda O(1). TreeMap → ordenado, O(log n). LinkedHashMap → mantiene orden de inserción.
Set: HashSet → unicidad, O(1). TreeSet → ordenado.
Map<String, Integer> precios = new HashMap<>();
precios.put("café", 350);
precios.getOrDefault("agua", 0);  // evita NPE
 
List<String> nombres = List.of("Ana","Bob"); // inmutable (Java 9+)
List<String> copia   = new ArrayList<>(nombres); // mutable
Usá List.of() y Map.of() para colecciones de solo lectura. Son más rápidas y seguras.
ver más ↓
java moderno
Lambdas & Streams
Procesar colecciones de forma declarativa. Reemplazan los loops imperativos por cadenas legibles.
filter · map · collect reduce · flatMap
Los Streams no modifican la colección original. Son lazy: no se evalúan hasta que hay una operación terminal (collect, count, findFirst). 
List<String> resultado = usuarios.stream()
    .filter(u -> u.getEdad() >= 18)
    .map(Usuario::getNombre)       // method reference
    .sorted()
    .collect(Collectors.toList());
 
// groupBy: muy útil en backend
Map<String, List<Pedido>> porCiudad = pedidos.stream()
    .collect(Collectors.groupingBy(Pedido::getCiudad));
Usá parallelStream() con cuidado: solo vale la pena con colecciones muy grandes y operaciones CPU-intensivas.
ver más ↓
java moderno
Optional, Records & Generics
Herramientas para código más seguro, expresivo y reutilizable.
Optional<T> record · <T extends>
Optional: evitá el == null explícito. Hacé visible en la firma que algo puede no existir. 
Optional<Usuario> user = repo.findById(id);
String nombre = user.map(Usuario::getNombre)
                    .orElse("Anónimo");
Records (Java 16+): clases inmutables de datos sin boilerplate. Perfectas para DTOs. 
record ProductoDTO(String nombre, double precio) {}
// genera: constructor, getters, equals, hashCode, toString
Generics: escribís código que funciona con cualquier tipo, con validación en compilación. 
public <T> ApiResponse<T> ok(T data) {
    return new ApiResponse<>(200, data);
}
ver más ↓
excepciones
Manejo de excepciones
Checked vs unchecked, try-with-resources y cómo diseñar excepciones propias.
try · catch · finally RuntimeException
Checked: el compilador obliga a manejarlas (ej: IOException). Unchecked: subclases de RuntimeException, no obligatorio manejarlas (ej: NullPointerException).

En backend moderno se prefiere unchecked para no contaminar las firmas de los métodos: 
// Excepción de dominio propia
public class UsuarioNoEncontradoException extends RuntimeException {
    public UsuarioNoEncontradoException(Long id) {
        super("Usuario no encontrado: " + id);
    }
}
 
// try-with-resources: cierra automáticamente
try (InputStream is = new FileInputStream("data.txt")) {
    // usar is
} // se cierra solo aunque haya excepción

Control de versiones distribuido. Fundamental para trabajar en equipo y para CI/CD.

runtime
JVM internals
Entender cómo la JVM gestiona memoria y ejecuta tu código te ayuda a depurar y optimizar.
heap · stack · GCbytecode · classloader
Heap: donde viven los objetos. Se divide en Young Generation y Old Generation. El Garbage Collector libera memoria automáticamente. Stack: frames de llamadas a métodos, variables locales.

Lo que necesitás saber para el día a día: OutOfMemoryError = el heap se llenó → revisá memory leaks. StackOverflowError = recursión infinita. El flag -Xmx512m limita el heap máximo a 512 MB.

java -Xms256m -Xmx512m -jar mi-app.jar

Bases de datos

relacional
SQL & PostgreSQL
El lenguaje universal para consultar bases de datos relacionales. PostgreSQL es el estándar de la industria.
SELECT · JOIN · INDEXACID · transacciones
Dominá: SELECT con JOIN, subconsultas, GROUP BY, ORDER BY, índices y transacciones. 
-- Query típica en backend
SELECT u.nombre, COUNT(p.id) AS total_pedidos
FROM usuarios u
LEFT JOIN pedidos p ON u.id = p.usuario_id
WHERE u.activo = true
GROUP BY u.id
HAVING COUNT(p.id) > 5
ORDER BY total_pedidos DESC;
Los índices aceleran las búsquedas pero enlentecen las escrituras. Ponelos en columnas que usás en WHERE y JOIN. Las transacciones garantizan ACID: Atomicity, Consistency, Isolation, Durability.
ver más ↓
ORM
JPA & Hibernate
Mapear objetos Java a tablas de base de datos. JPA es la especificación, Hibernate la implementación más usada.
@Entity · @RepositoryJPQL · Criteria API
@Entity
@Table(name = "usuarios")
public class Usuario {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
 
    @Column(nullable = false, unique = true)
    private String email;
 
    @OneToMany(mappedBy = "usuario", fetch = FetchType.LAZY)
    private List<Pedido> pedidos;
}
Cuidado con el N+1 problem: si cargás una lista y accedés a la relación de cada elemento, generás N consultas extra. Solución: @EntityGraph o JOIN FETCH en JPQL. Usá FetchType.LAZY por defecto.
ver más ↓
migraciones
Flyway & Liquibase
Control de versiones para el esquema de la base de datos. Imprescindible en equipos y CI/CD.
V1__init.sqlmigration scripts
En lugar de modificar la DB a mano, escribís scripts de migración versionados que se ejecutan automáticamente al iniciar la app.

    id BIGSERIAL PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    nombre VARCHAR(100) NOT NULL,

-- V2__agregar_columna_activo.sql
ALTER TABLE usuarios ADD COLUMN activo BOOLEAN DEFAULT true;
Spring Boot + Flyway se integran sin configuración extra. Los scripts nunca se modifican una vez ejecutados.

Diseño de APIs

protocolo
REST & HTTP
El estilo arquitectónico más usado para APIs web. Se basa en recursos, métodos HTTP y códigos de estado.
GET·POST·PUT·DELETE200·201·400·404·500
Reglas de oro de REST: los endpoints son sustantivos (no verbos), los métodos HTTP definen la acción. 
GET    /usuarios          → listar usuarios
GET    /usuarios/{id}     → obtener uno
POST   /usuarios          → crear (body con datos)
PUT    /usuarios/{id}     → reemplazar completo
PATCH  /usuarios/{id}     → actualizar parcial
DELETE /usuarios/{id}     → eliminar
Códigos importantes: 200 OK, 201 Created, 204 No Content, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 500 Internal Server Error.

Spring Framework

núcleo
Spring Core & DI
Inversión de control (IoC) e inyección de dependencias. Spring crea y gestiona tus objetos por vos.
@Component · @Bean@Autowired · @Qualifier
En vez de hacer new MiServicio(), Spring crea y provee los objetos (beans). Esto facilita el testing y desacoplamiento. 
@Service  // le dice a Spring que este es un bean
public class UsuarioService {
    private final UsuarioRepository repo;
 
    // inyección por constructor (preferida sobre @Autowired en campo)
    public UsuarioService(UsuarioRepository repo) {
        this.repo = repo;
    }
}
Scopes de beans: singleton (default, una instancia), prototype (nueva instancia cada vez), request / session (en apps web).

Spring Boot & MVC
Boot = Spring preconfigurado. MVC = patrón para construir APIs y apps web con controladores.

seguridad
Spring Security
Framework de seguridad para autenticación y autorización. Muy configurable.
@PreAuthorizeSecurityFilterChain
@Configuration
@EnableWebSecurity
public class SecurityConfig {
 
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(csrf -> csrf.disable())
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()

            .sessionManagement(s -> s.sessionCreationPolicy(STATELESS))
            .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class)
            .build();
    }

Mensajería & caché

async
Apache Kafka
Plataforma de streaming de eventos. Para comunicación asíncrona entre servicios y procesamiento de flujos.
producer · consumertopic · partition
Kafka almacena mensajes en topics divididos en partitions. Los producers publican, los consumers leen. Los mensajes persisten (no se eliminan al consumirse, a diferencia de RabbitMQ). 
// Publicar un evento
@Service
public class PedidoService {
    @Autowired KafkaTemplate<String, PedidoEvent> kafka;
 
    public void procesarPedido(Pedido pedido) {
        kafka.send("pedidos-creados", new PedidoEvent(pedido.getId()));
    }
}
 
// Consumir
@KafkaListener(topics = "pedidos-creados", groupId = "notificaciones")
public void onPedido(PedidoEvent event) {
    notificacionService.enviar(event.getPedidoId());
}
ver más ↓
caché
Redis
Base de datos en memoria ultra rápida. Úsala para cachear respuestas, sesiones y contadores.
@Cacheable · TTLString · Hash · Set
Redis guarda datos en memoria (microsegundos de latencia). Puede usarse con Spring Cache de forma transparente: 
@Service
public class ProductoService {
    // cachea el resultado. Si se llama de nuevo con el mismo id,
    // retorna del caché sin ir a la DB
    @Cacheable(value = "productos", key = "#id")
    public ProductoDTO findById(Long id) {
        return repo.findById(id).map(this::toDTO).orElseThrow();
    }
 
    // invalida el caché cuando se actualiza
    @CacheEvict(value = "productos", key = "#id")
    public void actualizar(Long id, ProductoDTO dto) { ... }
}
También usás Redis directamente para: rate limiting, locks distribuidos, pub/sub simple y sesiones compartidas entre instancias.
ver más ↓
mensajería
RabbitMQ
Message broker tradicional. Colas, exchanges y bindings para distribuir trabajo entre consumidores.
Queue · Exchange@RabbitListener
RabbitMQ es más simple que Kafka para casos de uso tradicionales: tareas en background, emails, notificaciones. El mensaje se elimina cuando se procesa. 
// Enviar tarea a la cola
rabbitTemplate.convertAndSend("emails-queue", new EmailTask(destinatario));
 
// Procesar en background
@RabbitListener(queues = "emails-queue")
public void procesarEmail(EmailTask task) {
    emailService.enviar(task.getDestinatario());
}
Elegí Kafka si necesitás replay de eventos, alta throughput o event sourcing. Elegí RabbitMQ para tareas simples en background y routing complejo de mensajes.

Testing

unitario
JUnit 5 & Mockito
JUnit para estructurar tests, Mockito para simular dependencias (mocks). La combinación más usada en Java.
@Test · @Mockwhen · verify · assert
@ExtendWith(MockitoExtension.class)
class UsuarioServiceTest {
 
    @Mock UsuarioRepository repo;
    @InjectMocks UsuarioService service;
 
    @Test
    void deberiaRetornarUsuarioCuandoExiste() {
        // Arrange
        var usuario = new Usuario(1L, "Ana", "ana@mail.com");
        when(repo.findById(1L)).thenReturn(Optional.of(usuario));
 
        // Act
        var resultado = service.findById(1L);
 
        // Assert
        assertThat(resultado.nombre()).isEqualTo("Ana");
        verify(repo, times(1)).findById(1L);
    }
}
Usá AssertJ en vez de las assertions de JUnit: son más legibles y tienen mejor autocompletado.

Concurrencia

threads
Threads & ExecutorService
Ejecutar tareas en paralelo. ExecutorService gestiona un pool de threads para no crear uno por tarea.
ExecutorServiceThreadPoolExecutor
Nunca uses new Thread() directamente en producción. Usá un pool: 
ExecutorService pool = Executors.newFixedThreadPool(10);
 
Future<String> future = pool.submit(() -> {
    // tarea que tarda
    return llamadaExterna();
});
 
String resultado = future.get(); // bloquea hasta tener resultado
pool.shutdown();
@Async de Spring es la forma más simple en una app Spring Boot: anota un método y se ejecuta en un thread del pool configurado.
ver más ↓
async moderno
CompletableFuture
API moderna para programación asíncrona y encadenar operaciones sin bloquear threads.
thenApply · thenComposeallOf · anyOf
// Llamadas en paralelo y combinar resultados
CompletableFuture<Usuario> userFuture = CompletableFuture
    .supplyAsync(() -> userService.findById(id));
 
CompletableFuture<List<Pedido>> pedidosFuture = CompletableFuture
    .supplyAsync(() -> pedidoService.findByUsuario(id));
 
// Esperar ambos y combinar
CompletableFuture.allOf(userFuture, pedidosFuture).join();
 
Usuario user = userFuture.get();
List<Pedido> pedidos = pedidosFuture.get();
// Ambas llamadas corrieron en paralelo ✓

reactive
Programación reactiva
Manejar streams de datos asíncronos con backpressure. Spring WebFlux y Project Reactor.
Mono · FluxWebFlux · R2DBC
Reactivo es útil cuando tenés muchas operaciones I/O concurrentes (miles de requests) y no querés bloquear threads. 
// Mono = 0 o 1 elemento. Flux = 0 o N elementos.
@GetMapping("/usuarios/{id}")
public Mono<UsuarioDTO> obtener(@PathVariable Long id) {
    return repo.findById(id)           // retorna Mono<Usuario>
               .map(this::toDTO)       // transforma
               .switchIfEmpty(Mono.error(new NotFoundException()));
}
Empezá con Spring MVC tradicional. Migrá a WebFlux solo si tenés un caso de uso claro (alta concurrencia I/O, streaming de datos).

Patrones de diseño

principios
SOLID
Cinco principios para diseñar código orientado a objetos mantenible y extensible.
SRP · OCP · LSPISP · DIP
S — Single Responsibility: cada clase tiene un único motivo para cambiar.
O — Open/Closed: abierto para extensión, cerrado para modificación (usá interfaces).
L — Liskov Substitution: podés usar una subclase donde se espera la clase padre sin romper nada.
I — Interface Segregation: interfaces pequeñas y específicas, no una enorme.
D — Dependency Inversion: dependé de abstracciones (interfaces), no de implementaciones concretas. 
// DIP en acción: el servicio depende de la interfaz, no de la clase concreta
public class NotificacionService {
    private final NotificacionSender sender; // interfaz
 
    // Podés inyectar EmailSender, SmsSender, etc. sin cambiar el servicio
    public NotificacionService(NotificacionSender sender) {
        this.sender = sender;
    }
}
ver más ↓
GoF
Patrones GoF más usados
Los patrones clásicos del Gang of Four que aparecen constantemente en proyectos Java.
Builder · FactoryStrategy · Observer
Builder: construir objetos complejos paso a paso (Lombok @Builder).

@Repository
public class UsuarioJpaAdapter implements UsuarioRepository {
